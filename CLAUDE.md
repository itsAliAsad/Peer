# Peer - P2P Academic Marketplace

## Quick Start

```bash
npm run dev          # Next.js dev server
npx convex dev       # Convex backend (separate terminal — both required)
npm run build        # Production build
npm run lint         # ESLint (flat config)
```

## Architecture

Peer is a P2P tutoring marketplace: students post help requests (buyers), tutors respond with offers (sellers). Next.js 16 App Router + Convex (reactive backend, no REST) + Clerk (auth). shadcn/ui (new-york style) + Tailwind CSS v4 + Framer Motion. State flows via `useQuery`/`useMutation` — no API layer.

Backend now follows a deeper-module split for cross-cutting concerns and domain entrypoints:
`convex/utils.ts` stays the stable auth/validation facade, while `identity.ts`, `audit.ts`, `trust.ts`, `notification_service.ts`, `conversations.ts`, and `tutor_profile_service.ts` hide shared backend policy. Public entry files such as `tickets.ts`, `offers.ts`, `messages.ts`, and `crash_courses.ts` should stay thin and delegate to workflow, policy, recommendation/ranking, and read-model modules.

## Critical Rules

- Every Convex mutation/query must call `requireUser()` or `requireAdmin()` from `convex/utils.ts`
- Never edit `convex/_generated/` — auto-generated by Convex codegen
- `app/template.tsx` runs on every navigation (Next.js template, not layout)
- `convex/tickets.ts`, `convex/offers.ts`, `convex/messages.ts`, and `convex/crash_courses.ts` should read like Convex facades, not mixed business-logic files
- Ticket recommendation/query orchestration belongs in `ticket_recommendations.ts`, `ticket_read_models.ts`, and `ticket_workflows.ts`
- Offer ranking/read shaping belongs in `offer_ranking.ts`, `offer_read_models.ts`, and `offer_workflows.ts`
- Message orchestration/read shaping belongs in `message_workflows.ts`, `conversation_read_models.ts`, and `conversations.ts`
- `convex/crash_courses.ts` is a stable Convex facade; keep heavy logic in `crash_course_policy.ts`, `crash_course_workflows.ts`, `crash_course_domain.ts`, `crash_course_enrollments.ts`, `crash_course_voting.ts`, `crash_course_read_models.ts`, or `crash_course_crons.ts`
- Notification writes should go through `convex/notification_service.ts`; payload validators live in `convex/notification_types.ts`
- Derived user trust state belongs in `convex/trust.ts` (reputation, verification tier)
- Components use `@/` path alias (maps to project root)
- `proxy.ts` at project root is a dev utility, not part of the app
